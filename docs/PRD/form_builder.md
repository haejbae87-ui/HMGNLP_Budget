# 교육신청양식마법사 요구사항 정의서 (PRD)

> **도메인**: 양식 마법사 및 동적 필드 관리 (Form Builder)
> **관련 파일**: `public/js/bo_form_builder.js`, `public/js/bo_form_logic.js`, `public/js/bo_forms.js`
> **최초 작성**: 2026-04-15
> **최종 갱신**: 2026-04-15
> **상태**: ✅ 완전 구현

## 1. 기능 개요
학습자(FO) 및 관리자(BO)가 교육 계획, 신청, 결과 등록 단계에서 사용할 입력 양식(Form Template)을 동적으로 설계하고 관리하는 기능입니다. 목적 및 서비스 유형(직접학습/교육운영)에 맞춰 입력 필드를 드래그 앤 드롭으로 구성하며, 필드 간 의존성 제어, 입력 권한(Scope) 설정, 버전 배포 및 보관 수명 주기를 제어합니다.

## 2. 사용자 스토리
> "플랫폼/테넌트 관리자는 서비스 목적에 맞는 양식을 생성하고, 필요한 입력 필드(예: 예상비용, 첨부파일 등)를 조립하여 배포할 수 있으며, 더 이상 FO에서 신규 사용을 막고자 할 때 양식을 보관(Archive) 처리할 수 있다."

## 3. 관련 DB 구조 및 메타데이터
양식 메타데이터와 필드 풀(Pool)은 DB 스키마와 클라이언트의 `ADVANCED_FIELDS` 캐시로 이원 관리됩니다.

- **`form_templates`**: 양식 마스터 스토어
  - `id` (PK, string), `tenant_id`, `domain_id` (가상조직 매핑), `account_code` (계정 매핑)
  - `type` (plan / apply / result), `purpose` (목적), `eduType` (교육유형)
  - `status` (draft / published / archived), `active` (boolean), `version` (int)
  - `fields` (JSON Array, 필드 키와 상태값 보관)
- **`form_field_catalog`**: L2 커스텀 필드 풀 보관
- **`field_options`**: select/multi_select 항목을 테넌트별 보관
- **`field_dependencies`**: 필드 간 선행 조건/종속성 룰셋
- **`tenant_l1_overrides`**: L1 공통 필드의 필수 여부/표시명 등을 테넌트별로 오버라이드

## 4. 화면별 기능 요구사항
| 탭/기능 명 | 세부 기능 및 설명 | 구현 상태 |
| --- | --- | --- |
| **양식 라이브러리** | 테넌트/가상조직/계정, 서비스 유형, 목적별 양식 목록 필터링 조회 | ✅ |
| **양식 복사** | 복사본 생성. `draft` 상태로 진입하며 정책 연결 정보는 제외됨 | ✅ |
| **양식 보관/복원** | `published` ↔ `archived` 토글. 보관 시 FO 화면에서 해당 양식 로딩 차단 | ✅ |
| **미리보기** | Front(학습자/권한자) 및 Back(승인자) 탭 분리 UI 렌더 시뮬레이터 | ✅ |
| **양식 편집기 (빌더)** | 좌측 필드 팔레트 클릭 추가 ↔ 우측 DnD 순서 변경 및 속성 편집 | ✅ |

## 5. 핵심 비즈니스 로직
- **보관(Archive) 방어 로직**: 
  양식이 하나 이상의 서비스 정책(Service Policy)에 연결되어 있을 경우, FO 서비스 중단을 방지하기 위해 경고 팝업이 노출됩니다. 보관 시 즉각 FO에서 신규 계획/신청 폼 생성이 불가능해집니다. (기존 데이터는 유지)
- **필드 의존성 자동 제어 (Predecessors)**: 
  예: '강사이력서' 필드는 '강사료' 필드가 존재해야만 정상 동작합니다. 빌더 UI에서 후행 필드를 추가할 경우, 정의된 선행 필드 배열(`predecessors`)을 자동 검사하여 누락된 필수 선행 필드를 강제 추가시킵니다. 삭제 시에도 연결된 필드가 있으면 차단합니다.
- **다중 입력 스코프 (Scope)**: 
  - `front`: FO 학습자/부서담당자에게 입력 폼 노출
  - `back`: FO 폼 렌더에서는 숨김, BO/결재자 전용 (오직 BO에서만 입력)
  - `provide`: BO 담당자가 입력하고, FO에서는 단순 Read Only 안내로 노출
  - `system`: 시스템 자체의 자동 렌더링 목적
- **배포 및 버전 갱신**: 
  초안(`draft`)에서 배포(`publish`) 시 양식의 버전을 자동 1 증가시킵니다. 기존 배포된 필드 항목이 삭제되어 데이터 유실 위험이 있을 경우, 렌더링 검수에 앞서 경고창을 통지합니다.

## 7. 예외 처리 및 방어 로직
- **양식 삭제 방어**: 연결된 서비스 정책이 하나라도 존재하면 삭제(Trash) 자체를 영구히 잠급니다. (버튼 disabled 처리 방식 적용 중)
- **이중 양식명 방어**: 이름 기반 중복 여부 확인 로직 내장 (새 양식 복사 시에도 경고 알림).
