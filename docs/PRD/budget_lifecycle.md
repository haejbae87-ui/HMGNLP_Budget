## 1. 기능 개요

교육예산의 **계획 → 수요예측 → 배정 → 사용 → 정산** 전 과정을 하나의 일관된 데이터 흐름으로 연결하는 시스템 고도화.
현재는 각 화면이 독립적으로 동작하여 "계획금액 → 배정금액 → 실사용금액"의 흐름이 끊어져 있으며,
예산 시뮬레이션, 조직개편 이관, 통장 간 이관 등 실무 핵심 기능이 부재.

### 2026-04-17 추가: 수요예측 묶음 상신 + 다단계 예산 배정

> [!IMPORTANT]
> **신규 비즈니스 요구사항**: 수요예측용 교육계획을 개별 상신이 아닌 **팀 단위로 묶어서 상신**하고,
> **운영담당자 → 총괄담당자** 다단계로 예산을 검토·배정하는 워크플로우가 필요.
> 이 기능은 **HMC/Kia 전용** (테넌트 feature flag 제어).

## 3. 상충 분석 — 기존 설계 vs 신규 요구사항

> [!WARNING]
> 아래 5개 영역에서 기존 PRD 설계와 신규 요구사항이 충돌하며, 설계 변경이 필요합니다.

### 3.1 예산 배정 흐름 충돌

| 항목 | 기존 설계 | 신규 요구사항 | 충돌 수준 |
|------|----------|-------------|:---:|
| **배정 주체** | 총괄담당자가 직접 시뮬레이션 → 확정 | 운영담당자 1차 배정 → 총괄 최종 확정 | 🔴 HIGH |
| **배정 단위** | 개별 교육계획 단위 | 묶음(bundle) 단위 → 내부 계획별 분배 | 🔴 HIGH |
| **시뮬레이션 사용자** | 총괄담당자 전용 | 운영담당자도 1차 시뮬레이션 필요 | 🟡 MED |

**해결 방향**: 시뮬레이션을 2단계로 분리
```
운영담당자 시뮬레이션 (1차 Envelope) → 총괄 시뮬레이션 (최종 Envelope)
```

### 3.2 결재라인 구조 충돌

| 항목 | 기존 설계 (approval_line_design.md) | 신규 요구사항 | 충돌 수준 |
|------|----------|-------------|:---:|
| **결재 대상** | 단건 교육계획 (plan) | 묶음 (bundle) | 🔴 HIGH |
| **결재 단계** | 금액 구간별 팀장→실장→본부장 | 팀장→운영담당자→운영팀장→총괄 (4단계 고정) | 🔴 HIGH |
| **결재 문서** | 단건 상세 | 요약 테이블 + 접이식 상세 | 🟡 MED |

**해결 방향**: 묶음 결재는 기존 결재라인과 **별도 채널**로 운영
- 기존 결재라인: 상시 교육계획 단건 결재 (유지)
- 묶음 결재라인: 수요예측 묶음 전용 (신규 `bundle_approval_routes` 테이블)

### 3.6 역할별 BO 화면 분리 vs 공유 분석 (2026-04-17 추가)

> [!IMPORTANT]
> **핵심 질문**: 운영담당자(hq_general/center_rnd)와 총괄담당자(total_general/total_rnd)가
> 같은 화면을 공유해야 하는가, 분리해야 하는가?

#### 현재 구조 (As-Is)

현재 `bo_layout.js` 분석 결과, 총괄(BUDGET_ADMIN_MENUS)과 운영(BUDGET_OP_MENUS)이
**동일한 메뉴**(plan-mgmt, budget-demand, allocation, budget-history)를 공유하며,
화면 내부에 **역할 기반 데이터 스코핑이 없음**. 운영담당자도 전체 조직 데이터를 볼 수 있는 상태.

#### 분석 결론: 화면 분리 X → 역할 기반 뷰 모드 O

화면 분리 최대 리스크: 코드 2벌 유지보수 = 버그 동기화 지옥. 따라서 **동일 화면에서 역할을 감지하여 데이터 범위와 기능 권한을 분기**하는 방식 채택.

| 화면 | 분리? | 총괄 | 운영 |
|------|:---:|------|------|
| 교육계획 관리 | X | 전체 + 배정편집 + 확정 | 관할만 + 검토승인 + 1차조정 |
| 예산 수요분석 | X | 전체 드릴다운 + Envelope + 확정 | 관할 드릴다운 + 1차 배분 |
| 예산 배정 | X | 전체 통장 + 배정/이관 | 관할 통장 + 조회/관할 이관 |
| 예산 사용이력 | X | 전체 | 관할만 (필터만 다름) |
| 교육신청 관리 | X | 전체 승인 | 관할 승인 |

#### 데이터 스코핑 핵심 로직

```javascript
// 공통 유틸: 현재 역할의 관할 조직 ID 목록
function _getManagedOrgIds() {
  const role = boCurrentPersona.role || '';
  const isTotal = /total|admin|platform/i.test(role);
  if (isTotal) return null; // null = 전체 (필터 없음)
  const groups = boCurrentPersona.managedGroups || [];
  const ids = [];
  groups.forEach(g => (g.teams||[]).forEach(t => ids.push(t.id||t.name)));
  return ids; // 관할 팀 ID 목록
}
```

```
┌────────────── FO (학습자/팀원) ──────────────────────┐

│                                                       │
│  팀 대표 → [내 팀 계획 묶기] → plan_bundles 생성       │
│         → 팀장에게 결재 요청                           │
│         → 결재문서: 팀 묶음 요약                        │
└───────────────────────────────────────────────────────┘
           │ 팀장 승인
           ▼
┌────────────── BO (운영담당자) ────────────────────────┐
│  관할 팀 묶음 수신 (교육조직 하위 팀들)                 │
│  1차 검토: 개별 교육계획별 예산 조정 (인라인 편집)      │

│                                                       │
│  [교육조직 묶음 생성] → 팀 묶음들을 2차 묶음             │

│  최종 검토: Envelope 설정 + 시뮬레이션 (기존 UI 재사용) │
│  → 최종 확정: plans.allocated_amount 일괄 반영          │
│  → 결과 통보: 운영담당자 → 팀장 → 작성자               │
```

### 7.1 plans 테이블 변경 (기존 + 신규)

```sql
  -- 소속 묶음 ID (수요예측 묶음에 포함된 경우)
```

### 7.2 plan_bundles (신규 테이블)

```sql
CREATE TABLE plan_bundles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id TEXT NOT NULL,
  bundle_type TEXT NOT NULL DEFAULT 'team_forecast',
    -- 'team_forecast' (팀 묶음) / 'org_forecast' (교육조직 묶음)
  parent_bundle_id UUID REFERENCES plan_bundles(id),
    -- 교육조직 묶음의 경우 상위가 없음, 팀 묶음은 교육조직 묶음의 하위
  title TEXT NOT NULL,
    -- 예: "2027년 수요예측 - 내구기술팀"
  fiscal_year INTEGER NOT NULL,
  org_id TEXT,             -- 팀 ID 또는 교육조직 ID
  org_name TEXT,           -- 조직명
  template_id TEXT,        -- virtual_org_template_id (제도그룹)
  creator_id TEXT,         -- 묶음 생성자 (팀 대표)
  creator_name TEXT,
  plan_count INTEGER DEFAULT 0,
  total_requested BIGINT DEFAULT 0,  -- 원래 요청 총액
  total_adjusted BIGINT DEFAULT 0,   -- 1차 조정 총액
  total_allocated BIGINT DEFAULT 0,  -- 최종 배정 총액
  status TEXT NOT NULL DEFAULT 'draft',
    -- draft → submitted → team_approved → ops_reviewing → ops_approved → 
    -- ops_leader_approved → final_reviewing → allocated → completed
  submitted_at TIMESTAMPTZ,
  approved_at TIMESTAMPTZ,
  allocated_at TIMESTAMPTZ,
  memo TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_plan_bundles_tenant_year ON plan_bundles(tenant_id, fiscal_year);
CREATE INDEX idx_plan_bundles_status ON plan_bundles(status);
```

### 7.3 bundle_adjust_log (묶음 내 예산 조정 이력)

```sql
CREATE TABLE bundle_adjust_log (

  original_amount BIGINT NOT NULL,
  adjusted_amount BIGINT NOT NULL,
  reason TEXT,
  adjusted_by TEXT,
  adjusted_at TIMESTAMPTZ DEFAULT NOW()
```

### 8.2 수요예측 묶음 상태 흐름 (신규)

```
팀 묶음:
  draft → submitted → team_approved → (운영담당자 수신)

교육조직 묶음:  
  draft → submitted → ops_approved → ops_leader_approved → (총괄 수신)
  → final_reviewing → allocated → completed
```

### 8.3 plan_type 자동분류 규칙 (신규)

```
교육 예정일 기준:
  예정일이 현재 연도 → plan_type = 'regular' (상시)
  예정일이 다음 연도+ → plan_type = 'forecast' (수요예측)
  예정일 미정 + 수요예측 기간 중 → plan_type = 'forecast'
  예정일 미정 + 수요예측 기간 외 → plan_type = 'regular'
```

### 8.4 묶음 결재 규칙 (신규 — approval_line_design.md와 분리)

```
묶음 결재는 기존 결재라인(2축 에스컬레이션)과 독립적으로 운영:

팀 묶음 결재:
  기안(팀 대표) → 승인(팀장) → [운영담당자 수신]

교육조직 묶음 결재:
  기안(운영담당자) → 승인(운영담당자 팀장) → [총괄담당자 수신]

※ 금액 기반 에스컬레이션(축1)과 soft_limit 초과(축2)는 
  개별 교육신청(apply) 시에만 적용. 수요예측 묶음에는 미적용.
```

### 개발 순서도 (업데이트)

```
P1 (DB ✅) ──┬──→ P16 (역할뷰 ❌) ──→ P2 (인라인편집 ❌) ──→ P3 (상세뷰 ❌)
             │         │
             │         └──→ P13 (운영검토 ❌) ──→ P14 (총괄배정 ❌)
             │
             ├──→ P4 (시뮬레이션 ✅) ──→ P5 (확정 🟡)
             │
             ├──→ P6 (bankbooks ✅) ─┬──→ P7 (통장이관 🟡)
             │                      ├──→ P8 (조직이관 ❌)
             │                      └──→ P9 (레포트 ❌) → P10 (실사용 ❌)
             │
             └──→ P11 (묶음기반 ❌) ──→ P12 (FO묶음 ❌) ──→ P15 (결재문서 ❌)
```

### 권장 구현 순서

> **1순위: Phase 16 (역할뷰)** — 모든 화면의 기반 데이터 스코핑. P2, P13의 선행 조건.
> **2순위: Phase 2 (인라인 편집)** — 운영담당자 1차 조정에도 재사용되는 핵심 UX.
> **3순위: Phase 11 → 12 (묶음 기반 → FO 묶음)** — 수요예측 핵심 워크플로우.
> **4순위: Phase 13 → 14 (운영검토 → 총괄배정)** — 다단계 예산 배정.
>
> ⚠️ **P16이 P2보다 먼저인 이유**: 인라인 편집 시 역할별 편집 가능/불가를
> 처음부터 반영해야 나중에 재작업이 없음.
