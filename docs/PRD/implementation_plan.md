# 폼 빌더 동기화 및 카테고리 개편 구현 계획 (V2 - 사용자 이미지 반영)

첨부해주신 엑셀 이미지 기반의 1depth 카테고리 개편안에 대해 도메인 전문가 위원회(Domain Council) 교차 검증을 진행했습니다.

---

## 🏛️ Domain Council (비즈니스 도메인 전문가 교차 검증)

**기획 요약**: 기존 5가지 1depth 카테고리를 세분화하여 **'기본정보 / 일정정보 / 대상자정보 / 장소정보 / 기타정보 / 첨부파일 / 비용정보'**의 7개 카테고리로 쪼개고, 중복되거나 불필요한 필드(위탁기관명, 계획상세내용, 안내사항 등)를 과감하게 삭제하며 일부 필드를 병합/추가(담당자 정보)하는 전면 구조 개편.

### 🕵️ 전문가 난상 토론

1. 🎓 **교육계획/수요예측 전문가 (Edu Plan Expert)**
   - **의견**: **찬성**. 입력 폼이 7개 섹션으로 나뉘어 스크롤이 더 길어질 수는 있지만, 정보의 성격이 명확히 구분되어 인지 부하가 줄어듭니다. 특히 "위탁기관명, 안내사항, 준비물" 등 불필요한 항목이 삭제되어 입력 피로도가 대폭 감소할 것입니다.

2. 🏢 **제도그룹/권한 전문가 (VOrg & Policy Expert)**
   - **의견**: **우려 및 보완 요구**. 
     - "담당자 정보 (추가, 유저 검색)": 해당 폼을 작성하는 사람이 본인이 아닌 다른 담당자를 대리 지정하는 기능이라면, 이는 결재 권한이나 예산 소유권(Ownership)과 직결됩니다. FO에서 '유저 검색'을 통해 담당자를 변경했을 때, 해당 예산의 차감 권한이 어떻게 넘어가는지에 대한 후속 로직이 필요합니다. 
     - "교육목표 / 교육내용 / 교육대상" 병합: 이미 대상자정보 섹션에 "교육대상"이 있는데 기본정보에서도 병합을 요구하셨습니다. 대상자정보의 "교육대상"과 성격이 겹치지 않도록 주의해야 합니다.

3. 💰 **예산/정산 전문가 (Budget & Ledger Expert)**
   - **의견**: **주의 요망**. 
     - 비용정보에 "고용보험 환급액" 필드가 추가되었습니다. 기존 시스템은 "환급 여부"만 체크하고 실제 환급액은 사후 정산 시점에 입력하거나 별도 관리했습니다. 계획/신청 단계에서 환급액을 입력하도록 하면 예산(Amount) 밸런싱 로직과 충돌할 수 있으니 `calc_grounds`와 어떻게 연동할지 정의가 필요합니다.

4. 📝 **폼/결재 거버넌스 전문가 (Form & Approval Expert)**
   - **의견**: **강한 비판적 우려 (Devil's Advocate)**.
     - **삭제되는 필드들**: "제공항목(안내사항, 준비물, 확정 교육장소)" 필드가 완전히 삭제되면, BO 관리자가 교육 확정 후 FO 사용자에게 안내를 전달할 창구가 사라집니다. 만약 다른 컴포넌트나 메일/알림톡으로 대체되었다면 괜찮지만, 폼 내에서 지워버리는 것이 맞는지 재확인해야 합니다.
     - **필드 병합**: "교육목표 / 교육내용 / 교육대상" 3개 필드를 물리적인 1개 DB 컬럼(textarea)으로 묶어버리면 나중에 통계나 리포트(엑셀 다운로드)에서 데이터를 분리해 낼 수 없습니다. UI 상에서만 합쳐 보이게 할지, 아니면 아예 하나의 `course_description` 필드로 통폐합할지 명확히 해야 합니다.

### 🚨 리스크 및 엣지 케이스
- **담당자 정보 (유저 검색)**: FO에 사용자 검색 컴포넌트를 붙이는 작업은 API 연동 및 컴포넌트 재사용성 확보가 필요합니다. 단순 텍스트 입력이 아니라면 공수가 발생합니다.
- **Provide 필드 삭제**: BO에서 관리자가 FO로 정보를 쏘아주던 `prov_guide`, `prov_materials` 등이 삭제되면 기존의 "신청 → 확정(관리자 안내)" 프로세스에 구멍이 생길 수 있습니다.

### 💡 통합 아키텍처 제안
1. **카테고리 7분할 수용**: 제안해주신 7개 카테고리로 FO/BO 구조를 완전히 맵핑합니다.
2. **필드 병합 (통폐합)**: "교육목표/내용/대상"은 1개의 큰 Textarea 필드(예: `learning_objective`)로 퉁치고 placeholder로 "[교육목표]\n[교육내용]\n[교육대상]"을 제시하여 작성하도록 유도합니다.
3. **삭제 요청 필드**: 과감하게 비활성화(Trim)하되, 기존 데이터는 DB에 보존합니다.

## Open Questions (Resolved)

> [!NOTE]
> 전문가 위원회 검토 결과 제기된 사항들은 다음과 같이 해결되었습니다.
> 
> 1. **"담당자 정보(유저 검색)" 필드**: `fo_form_loader.js` 에 임시 모의 유저 검색 팝업 로직(`_foOpenUserSearch`)을 연동하여 프론트엔드 UI를 구축했습니다. 추후 API 연동 시 해당 함수 내부 로직만 교체하면 작동합니다.
> 2. **"고용보험 환급액" 필드**: FO 입력창에 `ei_refund_amount` 항목을 추가하고 `is_ei_eligible` 이 true인 경우에만 입력할 수 있도록 렌더링 로직에 반영했습니다.
> 3. **제공항목(안내사항, 준비물 등) 삭제**: 백오피스(`bo_policy_builder.js`) 및 프론트오피스(`fo_form_loader.js`) 입력 양식에서 모두 제거하여 요구사항에 맞게 UI를 간소화했습니다. (기존 데이터 구조는 DB에서 유지됨)
