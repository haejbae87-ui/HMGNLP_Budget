## 1. 기능 개요

교육계획 관리 화면 안에 모달로 숨어있던 **수요예측 기간 설정** 기능을 독립 메뉴로 분리한다.
**제도그룹 단위**로 연도별 수요예측 접수 기간을 관리하며, LNB '교육지원제도 기준정보' 섹션에 별도 메뉴로 노출된다.

FO(Front Office)에서는 수요예측 기간이 설정된 제도그룹에 속한 예산계정에 한해, 해당 계정과 연결된 교육지원 운영 규칙(서비스 정책)을 통해 허용된 교육유형으로만 교육계획(수요예측)을 수립할 수 있도록 제어한다.

### 배경 및 분리 이유

| 항목 | 현재 문제 | 개선 방향 |
|---|---|---|
| 접근 경로 | 교육계획 관리 → 수요예측 기간 설정 버튼 | 독립 메뉴로 직접 접근 |
| 설정 단위 | 예산계정 단위 (세분화 과도) | 제도그룹 단위로 단순화 |
| 가시성 | 모달 안에 숨어있어 발견하기 어려움 | 좌측 메뉴에서 상시 접근 가능 |
| FO 연동 | account_code 단위로 기간 조회 | 제도그룹(vorg_template_id) 단위로 전환 |

## 2. 사용자 스토리

### BO (Back Office) — 관리자
> "교육 담당 관리자는 LNB 메뉴에서 **수요예측기간 관리**에 접속하여,
> 회사(테넌트)와 제도그룹을 선택한 후, 해당 연도의 수요예측 접수 시작일과 마감일을 설정할 수 있다."

### FO (Front Office) — 사용자
> "FO 사용자가 다음 연도 교육계획(수요예측) 수립을 시도할 때,
> 시스템은 자신이 속한 제도그룹의 수요예측 기간 설정 여부를 확인하고,
> 기간이 열려있을 때만 해당 제도그룹의 교육지원 운영 규칙에서 허용한 교육유형을 선택할 수 있다."

## 3. 상세 기능 요구사항

### 3.1 신규 메뉴 구조

```
교육지원제도 기준정보
├── 예산계정 관리
├── 교육양식 마법사
├── 세부산출근거 관리
├── 수요예측기간 관리  ← [신규]
└── 교육지원 운영 규칙
```

| 번호 | 기능 | 설명 | 우선순위 |
|---|---|---|---|

| F-003 | 기간 현황 카드 | 선택된 제도그룹의 수요예측 기간 상태 표시 | 🔴 HIGH |
| F-004 | 기간 설정/수정 | 시작일, 마감일 인라인 편집 + 저장 (upsert) | 🔴 HIGH |

| F-008 | 알림 배너 | 기간 미설정 제도그룹 경고 배너 | 🟡 LOW |

### 3.3 FO 화면 기능 (기존 로직 개선)

| 번호 | 기능 | 설명 | 우선순위 |
|---|---|---|---|
| F-101 | 제도그룹 기반 기간 조회 | `_checkForecastDeadline()` 함수를 `vorg_template_id` 기반으로 변경 | 🔴 HIGH |
| F-102 | 수요예측 버튼 활성 제어 | 기간 미설정/마감 시 "+ 교육계획 수립" 버튼 비활성화 또는 안내 메시지 표시 | 🔴 HIGH |
| F-103 | 허용 교육유형 필터링 | 수요예측 기간 중 제도그룹의 서비스 정책에서 허용한 edu_types만 Step 3에서 선택 가능 | 🔴 HIGH |

### 3.4 필터 디자인 기준 (예산계정 관리 참조)
- 행 1: `테넌트(회사)` 드롭다운 | `제도그룹` 드롭다운
- 행 2: `연도` 드롭다운
- 예산계정 관리(`bo_budget_account.js`)의 상단 필터바 스타일 일치 적용

### 3.5 교육계획 관리 화면 변경사항
- 기존 `📅 수요예측 기간 설정` 버튼 제거
- 기존 `🔒 일괄 마감` 버튼 제거
- 수요예측 요약 배너에 "→ 수요예측기간 관리에서 설정" 링크 추가

```sql

  ADD COLUMN IF NOT EXISTS vorg_template_id uuid
    REFERENCES virtual_org_templates(id) ON DELETE SET NULL;

-- 기존 레코드를 조회하기 위한 인덱스
```

## 5. 비즈니스 로직

### 5.1 기간 판별 로직 (현행 유지)
```
is_closed = true      → 🔒 수동마감
now < recruit_start   → ⏳ 접수전 (D-N)
now > recruit_end     → 🔴 기간만료
start <= now <= end   → 🟢 접수중 (D-N)
그 외                 → 🟢 접수중 (기간 무한)
```

### 5.2 FO 수요예측 진입 조건 (핵심 비즈니스 로직)
```
사용자 제도그룹(vorg_template_id) 확인
  ↓
해당 제도그룹의 forecast_deadlines 조회 (fiscal_year = next_year)
  ↓
기간 상태 판별:
  ├── status = 'open' → 진입 허용
  │     └── Step 3 교육유형 선택지 = 해당 제도그룹의 서비스 정책(account_codes 매칭)에서 허용된 edu_types
  ├── status = 'not_started' → 접수 전 안내 후 차단
  ├── status = 'expired' or 'closed' → 마감 안내 + ongoing으로 전환 확인
  └── null (기간 미설정) → '수요예측 기간이 설정되지 않았습니다' 안내 + ongoing 유도
```

### 5.3 FO 허용 교육유형 필터링 로직
```
제도그룹(vorg_template_id)
  → 해당 제도그룹의 budget_accounts 조회 (vorg_template_id 매핑)
  → 각 계정에 연결된 service_policies 조회
  → service_policies.edu_types 합집합이 허용 교육유형
  → plans.js Step 3에서 이 허용 교육유형만 표시
```

### 5.4 제도그룹 단위 적용 규칙
- 하나의 제도그룹은 하나의 연도별 수요예측 기간만 가질 수 있다
- 상시 계획(plan_type=ongoing)은 수요예측 기간에 무관하게 항상 허용
- 기간이 없는 제도그룹 → FO 수요예측 수립 불가 (ongoing만 가능)

## 7. 예외 처리 및 엣지 케이스

| 케이스 | 처리 방식 |
|---|---|
| 제도그룹이 0개인 테넌트 | "제도그룹이 없습니다. 먼저 제도그룹을 생성해주세요." 안내 |
| 시작일 > 마감일 설정 | "종료일은 시작일 이후여야 합니다" 인라인 validation |
| 연도가 과거인데 기간 설정 | 경고 toast 표시 후 저장 허용 (강제 불허 없음) |
| 이미 수요예측 계획이 접수된 상태에서 마감 | "N건의 계획이 접수 중입니다. 마감 시 추가 접수가 불가합니다." confirm |
| 동일 제도그룹+연도 중복 저장 | upsert 방식 (기존 레코드 업데이트) |
| DB 에러 | 에러 메시지 + 재시도 버튼 |
| FO 사용자가 제도그룹 매핑이 없는 경우 | 수요예측 기간 조회 skip, ongoing만 수립 가능 |
| 기간 조회 중 API 실패 | null 처리 → 기간 미설정으로 간주, 수요예측 차단 |
| 수요예측 기간 허용 교육유형이 0건인 경우 | "현재 수요예측 신청 가능한 교육유형이 없습니다" 안내 |
| 사용자가 복수의 제도그룹에 속한 경우 | 각 제도그룹별 계정을 개별로 처리 (어느 한 그룹의 기간이 열려있으면 진입 허용) |

## 8. 기획자 검토 필요 항목

> [!WARNING]
> **기존 `_checkForecastDeadline()` 로직 변경 영향**
> `plans.js` L597에서 `_checkForecastDeadline(tenantId, year)` 호출 시 account_code 없이 전체 조회 중.
> 제도그룹 단위로 전환 시 `currentPersona`의 VOrg 매핑(`vorgId`)이 반드시 있어야 함.
> VOrg 매핑이 없는 페르소나의 경우 기존 `__ALL__` 폴백 로직 유지 필요.

> [!IMPORTANT]
> **FO Step 3 교육유형 필터링이 현재 `getPolicyEduTypes()`에 의존**
> `plans.js`의 Step 3는 서비스 정책에서 edu_types를 필터링하는 `getPolicyEduTypes` 사용.
> 수요예측 기간 오픈 시 **이 정책 매칭이 제도그룹 기반으로 동작하는지** 확인 필요.
> 현재 `matchedPolicies`는 `vorgId` 기반으로 이미 필터링되고 있을 가능성이 있으므로 기존 로직 활용 가능.

> [!CAUTION]
> **복수 제도그룹 사용자의 교육유형 합집합 처리**
> 사용자가 여러 제도그룹에 속한 경우, 각 제도그룹의 기간과 허용 교육유형이 다를 수 있음.
> 일관된 UX를 위해 "기간이 열린 제도그룹의 교육유형만 허용"하는 교차 필터 정책 필요.

> [!NOTE]
> **알림 배지 제거 주의**
> `bo_layout.js` 또는 `bo_plan_mgmt.js`에서 pendingCount를 LNB 배지로 주입하는 로직 정확한 파악 후 제거.
> 다른 메뉴의 알림 배지 로직에 영향이 없는지 확인 필요.
