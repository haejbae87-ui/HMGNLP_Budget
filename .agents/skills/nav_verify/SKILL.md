# 내비게이션 및 상호작용 전수 검증 스킬

본 스킬은 모든 화면 개발 시 **'내비게이션 무결성 검토'**를 필수로 수행합니다.
화면 구현 후 코드를 제출하기 전에 반드시 아래 3단계를 거쳐야 합니다.

---

## 1단계: 라우팅 및 링크 전수 조사 (Static Analysis)

### 검증 항목
- 작성된 모든 **onclick 핸들러, boNavigate() 호출, 라우팅 함수**의 경로/메뉴ID가 실제 구현된 함수와 일치하는지 확인
- 동적 파라미터(ID 등)의 경우, 데이터가 없을 때 에러가 발생하지 않는지 방어 코드 체크
- 메뉴 배열(`PLATFORM_MENUS`, `TENANT_ADMIN_MENUS` 등)에 새 메뉴가 추가되면 **반드시** `boNavigate()`의 if 분기와 `accessMenus` 배열에도 추가했는지 확인

### 실행 방법
```powershell
# 1. 신규 메뉴 ID가 라우팅에 등록되었는지 확인
Select-String -Path "public\js\bo_layout.js" -Pattern "menuId" | Select-Object LineNumber, Line

# 2. ⚠️ [필수] 메뉴 ID가 Supabase DB에 모두 등록됐는지 자동 비교 (dry-run)
node ".agents\skills\auto_deploy\check_menu_db_sync.js"

# 누락 항목이 있으면 자동 INSERT
node ".agents\skills\auto_deploy\check_menu_db_sync.js" --fix
```

> **⚠️ 핵심 주의사항**: `checkMenuAccess()` 함수는 Supabase `role_menu_permissions` 테이블을 **1순위**로 참조합니다.
> `bo_data.js` `accessMenus`는 DB 로드 실패 시만 사용되는 **폴백**입니다.
> 따라서 메뉴를 코드에 추가하면 **반드시 DB에도 INSERT** 해야 합니다.

### 체크리스트
- [ ] 메뉴 배열에 추가한 ID가 `boNavigate()` 함수에 라우팅 분기 존재
- [ ] 해당 렌더 함수가 실제로 정의되어 있고, `backoffice.html` (또는 `frontoffice.html`)에 script 태그 포함
- [ ] 페르소나 `accessMenus` 배열에 새 메뉴 ID 추가 완료
- [ ] **`role_menu_permissions` DB 테이블에 신규 메뉴 ID 등록** (`check_menu_db_sync.ps1` 실행으로 검증) ← **★ 최다 누락 항목**
- [ ] 동적 데이터 의존 시 null/undefined 방어 코드 존재

## 2단계: 브라우저 에이전트 실행 (Dynamic Testing)

### 필수 수행 항목
코드 작성 완료 후, **반드시** `browser_subagent` 도구로 배포된 화면에 접속하여 다음을 전수 검증합니다:

1. **개발한 화면에 접속** (배포 URL 또는 로컬 서버)
2. **화면에 노출된 모든 메뉴 버튼, GNB(상단바), 사이드바 링크를 하나씩 클릭**
3. 각 클릭 시 확인할 사항:
   - ✅ URL이 정상 변경되는가
   - ✅ 콘솔(Console)에 에러가 뜨지 않는가
   - ✅ 'Page Not Found' 또는 빈 화면이 나오지 않는가
   - ✅ 의도한 콘텐츠가 렌더링되는가

### 브라우저 에이전트 호출 예시
```
browser_subagent:
  Task: |
    1. https://haejbae87-ui.github.io/HMGNLP_Budget/backoffice.html 접속
    2. 사이드바에서 [새로 추가된 메뉴명] 클릭
    3. 화면 렌더링 확인 + 콘솔 에러 확인
    4. 주요 버튼/링크 클릭 테스트
    5. 결과 보고: 성공/실패, 에러 메시지
```

## 3단계: 상태 유지 및 데이터 로딩 검증

### 검증 항목
- 메뉴 이동 시 **Supabase 데이터 로딩** 중 화면이 멈추거나 하얗게 변하지 않는가 (Loading State)
- **뒤로가기 버튼** 클릭 시 이전 상태가 잘 유지되는가
- 데이터가 없는 경우 **빈 상태(Empty State)** UI가 정상 표시되는가
- DB 에러 발생 시 **에러 핸들링 UI**가 표시되는가 (try/catch fallback)

### 체크리스트
- [ ] loading 상태에서 적절한 피드백 제공 (로딩 스피너 또는 스켈레톤)
- [ ] 데이터 0건일 때 빈 상태 안내 메시지 표시
- [ ] DB 에러 시 사용자 친화적 에러 메시지 + 재시도 버튼
- [ ] 뒤로가기/메뉴 재진입 시 이전 상태 유지 또는 정상 재로딩

## 🔥 스킬 발동 조건

이 스킬은 다음 상황에서 **자동으로** 실행됩니다:

1. **새 메뉴/페이지 추가** 작업 완료 시
2. **라우팅 변경** (메뉴 배열, boNavigate 등) 작업 완료 시
3. **기존 화면의 상호작용 요소** (버튼, 링크, 탭 등) 변경 시
4. 사용자가 "테스트해줘", "검증해줘", "확인해줘" 등 요청 시
