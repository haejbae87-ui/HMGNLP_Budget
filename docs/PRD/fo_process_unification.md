# FO 프로세스 통합 UX 요구사항 정의서 (PRD)

> **도메인**: FO UX 프로세스 통합
> **관련 파일**: `fo_apply_list.js`, `fo_apply_form.js`, `fo_apply_actions.js`, `fo_plans_list.js`, `bo_budget_account_mgmt.js`
> **최초 작성**: 2026-05-10
> **최종 갱신**: 2026-05-10
> **상태**: 🟡 기획 확정 / 구현 대기

## 1. 기능 개요

4개 FO 모듈(사업계획, 운영계획, 교육신청, 결과등록)의 진입 프로세스를 **"제도그룹 → 예산계정 → 목록 → 위저드(3단계)"**로 통일합니다. 기존 사업계획/운영계획의 허브 패턴을 교육신청/결과등록에 이식하는 리팩토링입니다.

## 2. 사용자 스토리

> "교육 담당자는 교육신청 메뉴에 진입할 때, 사업계획/운영계획과 동일하게 제도그룹과 예산계정을 먼저 선택한 후 해당 계정의 신청 목록을 확인하고 새 신청을 시작할 수 있다."

> "개인직무 사외학습 사용자가 VOrg 1개·계정 1개에 소속되어 있으면, 허브 화면 없이 바로 목록 화면으로 자동 이동된다."

> "무예산 계정(`uses_budget=false`)은 계정 허브에서 다른 계정과 동일하게 카드로 표시되며, 선택 시 위저드에서 비용 필드가 자동 비활성화된다."

> "관리자는 BO 예산계정 설정에서 `list_view_mode`를 설정하여, FO 목록에서 개인 탭만/팀 탭만/둘 다 보이도록 제어할 수 있다."

## 3. 확정 결정사항

| 항목 | 결정 | 근거 |
|---|---|---|
| Q1: 탭 노출 | 탭 유지 + `budget_accounts.list_view_mode` 설정 (`personal`/`team`/`both`) | 계정별로 관리자가 제어 |
| Q2: 무예산 경로 | 별도 카드 불필요 — `uses_budget=false` 계정이 허브에 자연 노출 | PRD(`service_policy.md`, `form_simplification.md`) 확인 완료 |
| Q3: 배포 전략 | 일괄 배포 | 리팩토링 수준 (기존 패턴 재사용) |

## 5. 자동 스킵 규칙

| 조건 | 동작 | 추가 클릭 |
|---|---|---|
| VOrg 1개 | 자동 선택, 계정 허브로 | 0 |
| VOrg 2개+ | VOrg 선택 카드 표시 | +1 |
| 계정 1개 | 자동 선택, 목록으로 | 0 |
| 계정 2개+ | 계정 선택 카드 표시 | +1 |

> **대다수 사용자(VOrg 1개 + 계정 1개)는 현재와 완전히 동일한 UX. 추가 클릭 0.**

## 6. DB 변경

## 7. 핵심 비즈니스 로직

### 계정 컨텍스트 자동 주입
- 허브에서 선택된 계정 코드를 `applyState.budgetId`에 자동 매핑
- 위저드에서 예산 선택 Step 제거 → 계정은 이미 확정 상태

### 무예산 처리 (기존 로직 그대로)
- `uses_budget=false` 계정 선택 시 → 위저드에서 비용 필드 전체 disable
- `fo_form_loader.js`의 기존 `isNoBudget` 판별 로직 유지

### 탭 노출 제어
- `budget_accounts.list_view_mode` 값을 FO에서 읽어 탭 표시/숨김
- `teamViewEnabled`(페르소나 기반)과 AND 조건으로 결합

## 9. 예외 처리 및 엣지 케이스
| 케이스 | 처리 |
|---|---|
| VOrg 0개 | `default` VOrg 자동 할당, 계정 허브로 직행 |
| 계정 0개 | "이 제도그룹에 배정된 예산계정이 없습니다" 안내 |
| 교육계획 기반 신청 (`_applyFromPlan`) | 계획에 포함된 `account_code`로 허브 자동 스킵 |
| `list_view_mode` 미설정 | 기본값 `'both'` → 기존 동작과 동일 |
