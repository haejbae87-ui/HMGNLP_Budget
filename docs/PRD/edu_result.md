# 교육결과 등록 요구사항 정의서 (PRD)

> **도메인**: 교육결과 등록 (Education Result Registration)
> **관련 파일**: `result.js` (신규), `gnb.js`, `frontoffice.html`, `utils.js`, `apply.js`
> **최초 작성**: 2026-04-08
> **최종 갱신**: 2026-04-08
> **상태**: 🔴 미구현

---

## 1. 기능 개요

교육 이수 후 결과를 등록하는 **독립 화면**을 신규 구축한다.
현재 교육신청(apply.js) 안에 결과 등록 위저드가 혼재되어 있어 UX가 혼란스럽고,
교육계획·교육신청 위저드의 Step1 목적선택에 "결과만 등록"이 섞여 표시되는 문제를 해결한다.

**핵심 원칙**: 패턴 A/B(신청 기반)와 패턴 C/D(직접 등록)를 **하나의 화면**에서 분기 처리하되,
교육계획·교육신청 화면에서는 결과 관련 항목을 완전히 제거한다.

## 2. 사용자 스토리

> "학습자는 **교육결과 등록** 메뉴에서 이수한 교육의 결과를 등록할 수 있다."
> - 패턴 A/B 정책: 승인완료된 교육신청을 선택한 뒤 결과를 입력한다.
> - 패턴 C/D 정책: 교육신청 없이 교육 정보를 직접 입력하여 결과를 등록한다.

| F-004 | 신청 기반 결과 (A/B) | 승인완료된 교육신청 목록에서 선택 → 결과 입력 위저드 | 🔴 HIGH |

## 4. DB/데이터 구조

### 기존 테이블 활용: `applications`

| 컬럼 | 용도 |
|---|---|
| `id` | PK (결과: `RES-{timestamp}`) |
| `status` | 결과 등록 시 `completed` 또는 `result_pending` |
| `plan_id` | 패턴 A: 연계된 교육계획 ID |
| `policy_id` | 매칭된 서비스 정책 ID |
| `detail` (JSONB) | `resultType: 'from_application' | 'direct'`, 결과 데이터 |

> ⚠️ 신규 테이블 불필요: applications 테이블의 `detail.resultType` 필드로 분기

## 5. 비즈니스 로직

### 5.1 패턴 분기 판단

```
매칭된 정책 탐색:
- process_pattern IN ('A','B') → "신청 기반 결과 등록" 옵션 표시
- process_pattern IN ('C','D') → "직접 결과 등록" 옵션 표시
- 양쪽 모두 있으면 → 두 옵션 모두 표시
```

### 5.2 신청 기반 결과 등록 (패턴 A/B)

1. 승인완료 상태(`approved`)의 교육신청(applications) 목록 표시
2. 교육신청 선택 → 교육 정보 자동 채움
3. 결과(수료 여부, 소감, 실제 비용 등) 입력
4. DB 저장: 해당 application의 status를 `completed`로 업데이트 + detail에 결과 데이터 병합

### 5.3 직접 결과 등록 (패턴 C/D)

1. 교육 정보 직접 입력 (과정명, 기간, 기관 등)
2. 패턴 C: 비용 정보 입력 (후정산) / 패턴 D: 비용 없음
3. 결과 작성 (소감, 첨부 등)
4. DB 저장: 새 application row INSERT (status: 'completed')

## 6. 접근 권한

| 역할 | 권한 |
|---|---|
| 학습자 (모든 사용자) | 본인 결과 등록/조회 |

## 7. 예외 처리 및 엣지 케이스

| 케이스 | 처리 방식 |
|---|---|
| 승인완료 신청이 0건 | "승인완료된 교육신청이 없습니다" 안내 + 직접등록 유도 |
| 이미 결과가 등록된 신청 | 목록에서 "결과 제출 완료" 뱃지 표시, 중복 등록 방지 |
| 정책 매칭 0건 | "교육결과 등록이 가능한 정책이 없습니다" 경고 |
| C/D 정책만 있는 사용자 | "직접 결과 등록"만 표시, 신청 기반 옵션 숨김 |
| A/B 정책만 있는 사용자 | "신청 기반 결과 등록"만 표시, 직접등록 옵션 숨김 |

## 8. [기획자 검토 필요 항목]

> [!WARNING]
> **기존 apply.js 결과 등록 코드 처리**: 현재 apply.js에 `_renderResultForm`, `_resultState`,
> `submitResult`, `_applySmartButtons` 등 결과 등록 로직이 존재. 신규 result.js로 이관 후
> apply.js에서 해당 코드를 비활성화 또는 result.js로 리다이렉트 처리 필요.

> [!IMPORTANT]
> **결과 양식(form_templates) 연동**: BO 서비스 정책에 `resultFormId`가 설정된 경우
> 동적 양식(renderDynamicFormFields) 렌더링 지원 필요. 현재는 하드코딩된 필드만 사용 중.

> [!NOTE]
> **패턴 E(자유선택)**: 현재 정의상 E는 "자유 선택"이나, 실제 DB에 E 패턴 정책이 없음.
> 향후 추가 시 A/B와 동일하게 신청 기반 결과 등록으로 처리하면 됨.
